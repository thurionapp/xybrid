# Releases & verification

This document explains how Xybrid is released and how you can verify that a
release is authentic and untampered before depending on it.

## How releases are made

Xybrid follows [Semantic Versioning](https://semver.org/). Each release is a
Git tag of the form `v<MAJOR>.<MINOR>.<PATCH>` (e.g. `v0.1.1`) and is published
from `master` through an automated two-step pipeline
([`release-prep.yml`](.github/workflows/release-prep.yml) →
[`release-publish.yml`](.github/workflows/release-publish.yml)).

The same version is published across all distribution channels:

| Channel        | Package                                                          |
|----------------|-----------------------------------------------------------------|
| crates.io      | [`xybrid`](https://crates.io/crates/xybrid)                      |
| pub.dev        | [`xybrid_flutter`](https://pub.dev/packages/xybrid_flutter)     |
| Maven Central  | [`ai.xybrid:xybrid-kotlin`](https://central.sonatype.com/artifact/ai.xybrid/xybrid-kotlin) |
| Swift PM       | this repository (tag / `swift` branch)                          |
| GitHub Releases| CLI binaries, XCFramework, Android libraries                    |

## Verifying integrity and authenticity

### Build provenance (SLSA attestations)

The release binaries attached to GitHub Releases — the XCFramework
(`XybridFFI-<tag>.xcframework.zip`), the CLI binaries (`xybrid-<tag>-<platform>`),
and the Android libraries — are published with
[SLSA build provenance attestations](https://docs.github.com/actions/security-guides/using-artifact-attestations-to-establish-provenance-for-builds)
generated by GitHub Actions. The attestation cryptographically proves that the
artifact was built by this repository's official release workflow.

Verify a downloaded artifact with the [GitHub CLI](https://cli.github.com/):

```sh
gh attestation verify <downloaded-file> --repo xybrid-ai/xybrid
```

A successful run confirms the artifact's provenance points at
`xybrid-ai/xybrid` and was produced by the release workflow. If verification
fails, do **not** use the artifact.

### Swift package checksum

For the Swift Package Manager, `Package.swift` pins the binary XCFramework by
its SHA-256 checksum (`xybridFFIChecksum`). Swift PM refuses to use the binary
if the downloaded archive does not match, so integrity is enforced
automatically when you resolve the package.

### Package registries

crates.io, pub.dev, and Maven Central serve packages over authenticated HTTPS
and record per-package checksums; their tooling verifies integrity on install.
Always depend on a pinned version and let your lock file (`Cargo.lock`,
`pubspec.lock`, etc.) record the resolved checksum.

## Reporting a problem

If you believe a published release has been tampered with, or you find a
security issue, follow [SECURITY.md](SECURITY.md). Please do **not** open a
public issue for security reports.
